# Автоматический деплой проекта в GitHub и Vercel

Этот документ описывает, какие шаги нужно выполнить агенту, чтобы:

- создать репозиторий в GitHub для текущей папки с git‑репозиторием;
- запушить код;
- выдать GitHub‑репозиторий установленному GitHub App Vercel;
- создать или привязать Vercel‑проект к этому репозиторию;
- дождаться успешного деплоя и проверить доступность сайта.

Все шаги должны выполняться автоматически, без действий пользователя в браузере.

---

## Предварительные условия

1. В текущей директории уже инициализирован git‑репозиторий.
2. Настроен git‑пользователь (user.name, user.email).
3. В `.env` (см. отдельный файл) заданы все необходимые переменные:
   - `GITHUB_USERNAME`
   - `GITHUB_TOKEN`
   - `GITHUB_APP_INSTALLATION_ID`
   - `VERCEL_TOKEN`
4. GitHub App Vercel уже установлен на аккаунт, в режиме **Only select repositories**.
5. У агента есть возможность:
   - выполнять команды shell (`git`, `curl` или HTTP‑клиент);
   - читать и парсить JSON‑ответы API;
   - читать этот файл и следовать шагам последовательно.

---

## Шаг 0. Preflight (обязателен)

Перед шагами 1+ агент должен выполнить preflight в текущей рабочей директории проекта:

1. Не менять рабочую директорию проекта (работать строго в `cwd`, откуда вызвали команду).
2. Предпочтительно запускать универсальный runner `scripts/deploy.py` (Python 3); если Python недоступен — использовать `scripts/deploy.js` (Node.js). Если оба рантайма недоступны — выполнять шаги вручную shell/API.
3. Проверить, что git доступен и есть доступ на запись в `.git` (или в директорию проекта, если `.git` еще нет).
4. Проверить доступность API:
   - `https://api.github.com` (всегда),
   - `https://api.vercel.com` (для `/deploy all`).
5. Если в окружении выставлены нерабочие proxy-переменные, запускать API и git-команды с очищенными proxy-переменными.
6. Если preflight не пройден — остановиться и вернуть понятную техническую причину.

## Часть 1. Публикация локального репозитория в GitHub

### Шаг 1. Определить имя проекта

1. Получи имя текущей директории.
2. Нормализуй его до валидного имени репозитория GitHub:
   - только буквы, цифры, дефисы и подчёркивания;
   - без пробелов.
3. Если нужно, запиши выбранное имя как `REPO_NAME` в памяти/контексте.

### Шаг 2. Проверить статус git

1. Убедись, что текущая папка — git‑репозиторий:
   - если `git rev-parse --is-inside-work-tree` возвращает ошибку — выполнить `git init`.
2. Убедись, что есть хотя бы один коммит:
   - если `git log` выдаёт ошибку, то:
     - добавить все файлы: `git add .`
     - создать первый коммит: `git commit -m "Initial commit"`

### Шаг 3. Создать репозиторий в GitHub через REST API

1. Отправь запрос:

   - Метод: `POST`
   - URL: `https://api.github.com/user/repos`
   - Заголовки:
     - `Authorization: token ${GITHUB_TOKEN}`
     - `Accept: application/vnd.github+json`
   - Тело (JSON):

     ```json
     {
       "name": "REPO_NAME",
       "private": true,
       "auto_init": false
     }
     ```

2. Если репозиторий уже существует и API вернул ошибку `"name already exists on this account"`, можно:
   - либо использовать существующий репозиторий с этим именем;
   - либо добавить суффикс к имени (`-1`, `-2` и т.п.) и повторить запрос.

3. Сохрани из ответа:
   - `clone_url` (HTTPS URL репозитория),
   - `ssh_url` (если будешь использовать SSH),
   - `id` (это `GITHUB_REPOSITORY_ID`),
   - `full_name` (`owner/name`).

### Шаг 4. Прописать удалённый репозиторий и запушить

1. Настрой origin:

   - Если origin ещё не настроен:
     - `git remote add origin <clone_url>`
   - Если origin уже есть, но указывает на другой репозиторий:
     - обнови: `git remote set-url origin <clone_url>`

2. Убедись, что основная ветка называется `main`:
   - если нет ветки main, создай её: `git branch -M main`

3. Запушь код:
   - `git push -u origin main`

4. Считай URL репозитория GitHub для человека:
   - `https://github.com/OWNER/REPO_NAME`

## Часть 2. Выдать репозиторий GitHub App Vercel через GitHub API

Цель: автоматически добавить только что созданный репозиторий в список репозиториев, доступных установленному GitHub App Vercel.

### Шаг 5. Получить `repository_id`, если он ещё не сохранён

Если на шаге 3 не был сохранён `id` репозитория:

1. Вызови:

   - Метод: `GET`
   - URL: `https://api.github.com/repos/OWNER/REPO_NAME`
   - Заголовки:
     - `Authorization: token ${GITHUB_TOKEN}`
     - `Accept: application/vnd.github+json`

2. Возьми поле `id` из ответа и запомни как `GITHUB_REPOSITORY_ID`.

### Шаг 6. Добавить репозиторий к установке GitHub App Vercel

1. Используй уже известный `GITHUB_APP_INSTALLATION_ID` (берётся из `.env`).
2. Вызови:

   - Метод: `PUT`
   - URL:  `https://api.github.com/user/installations/${GITHUB_APP_INSTALLATION_ID}/repositories/${GITHUB_REPOSITORY_ID}`
   - Заголовки:
     - `Authorization: token ${GITHUB_TOKEN}`
     - `Accept: application/vnd.github+json`

3. Ожидаемый успешный код ответа: `204 No Content`.
4. В случае ошибки:
   - если код 404 или сообщение о недостатке прав:
     - не пытаться продолжать деплой;
     - сообщить пользователю, что нужно проверить установку GitHub App Vercel и права токена.

## Часть 3. Создание/привязка проекта Vercel и деплой

Предполагается, что Vercel уже авторизован в GitHub и может обращаться к репозиториям, на которые установлен GitHub App.

### Общие параметры Vercel API

1. Все запросы к Vercel должны содержать заголовок:
   - `Authorization: Bearer ${VERCEL_TOKEN}`
   - `Content-Type: application/json`
2. Базовый URL API: `https://api.vercel.com`.

Если указан `VERCEL_TEAM_ID`, все запросы выполняются от имени команды с параметром `?teamId=${VERCEL_TEAM_ID}`.

### Шаг 7. Проверить, существует ли проект в Vercel

1. Имя проекта Vercel возьми такое же, как `REPO_NAME` (можно нормализовать).
2. Вызови:

   - Метод: `GET`
   - URL: `https://api.vercel.com/v9/projects/${REPO_NAME}?teamId=${VERCEL_TEAM_ID}` (параметр `teamId` добавляй только если он задан).

3. Интерпретация ответа:
   - Если 200 и приходит описание проекта — проект уже существует, используй его `id` как `VERCEL_PROJECT_ID` и переходи к шагу 9.
   - Если 404 — проекта нет, переходи к шагу 8.

### Шаг 8. Создать проект Vercel, привязанный к GitHub‑репозиторию

1. Сформируй JSON‑тело:

   ```json
   {
     "name": "REPO_NAME",
     "framework": null,
     "gitRepository": {
       "type": "github",
       "repo": "OWNER/REPO_NAME"
     }
   }
   ```

   - `framework` можно оставить `null`, чтобы Vercel сам определил фреймворк,
     либо задать, например "nextjs" / "vite" и т.п.

2. Вызови:

   - Метод: `POST`
   - URL: `https://api.vercel.com/v9/projects?teamId=${VERCEL_TEAM_ID}` (если `VERCEL_TEAM_ID` пуст, не добавляй параметр).

3. Из успешного ответа (201) запомни:
   - `id` проекта (`VERCEL_PROJECT_ID`),
   - `name` (он же `REPO_NAME`),
   - `link.type` и `link.org` / `link.repo` — для отладки.

Если ответ содержит ошибку о недоступности репозитория, значит GitHub App Vercel не имеет доступа к этому репо — в этом случае нужно пересмотреть шаг 6.

## Часть 4. Запуск деплоя и проверка

В большинстве случаев достаточно просто пуша в основную ветку — Vercel сам создаст деплой на основании webhooks. Однако для явного контроля можно дополнительно создать деплой через API.

### Шаг 9. Дождаться деплоя после пуша

1. Сразу после пуша в GitHub сделай небольшую задержку (например, 10–30 секунд), чтобы вебхук GitHub → Vercel успел обработаться.
2. Затем опрашивай список деплоев проекта:

   - Метод: `GET`
   - URL:  `https://api.vercel.com/v6/deployments?projectId=${VERCEL_PROJECT_ID}&limit=1&teamId=${VERCEL_TEAM_ID}`

3. Из ответа возьми первый (последний) деплой:
   - `deployment_id` — `id`,
   - `deployment_url` — `url`.

4. Если деплоев пока нет, повторяй запрос раз в 10–20 секунд, но не дольше нескольких минут.

### Шаг 10. Ожидание статуса деплоя

1. Опрашивай конкретный деплой:

   - Метод: `GET`
   - URL:  `https://api.vercel.com/v13/deployments/${deployment_id}?teamId=${VERCEL_TEAM_ID}`

2. Анализируй поле `readyState`:
   - если `readyState` = "READY" — деплой завершён успешно;
   - если `readyState` = "ERROR" или "CANCELED" — деплой провалился;
   - иначе (например, "BUILDING", "QUEUED") — подожди 10–20 секунд и повтори запрос.

3. Если через разумное число попыток деплой не завершился, сообщи пользователю о тайм‑ауте и приложи лог‑URL (`inspectorUrl` или аналогичное поле, если доступно).

### Шаг 11. Проверка доступности сайта

1. Как только `readyState` = "READY", возьми `deployment_url`.
2. Выполни HTTP‑запрос:

   - Метод: `GET`
   - URL: `https://${deployment_url}` (если `deployment_url` уже содержит протокол, используй как есть).

3. Ожидаемый результат проверки:
   - код ответа 200–299;
   - ответ содержит хотя бы некоторый HTML (например, наличие `<html` или `<title` в теле).
4. В случае успеха сообщи пользователю финальный URL деплоймента и, если есть, production‑URL проекта (можно получить из описания проекта Vercel).

---

## Команда пользователя

Когда пользователь говорит: "Опубликуй сайт", агент должен:

1. Пройти шаги 1–4 для публикации кода в GitHub.
2. Пройти шаги 5–6 для выдачи репозитория GitHub App Vercel.
3. Пройти шаги 7–11 для создания/обновления проекта Vercel и проверки деплоя.

Пользователь не должен ничего делать в интерфейсах GitHub или Vercel.
